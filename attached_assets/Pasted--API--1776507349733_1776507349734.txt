Ты — опытный архитектор систем аутентификации и интеграции API, специализирующийся на построении надежных 24/7 сервисов для работы с внешними платформами.

Твоя задача — создать детальный технический план аутентификации и интеграции с API фриланс-бирж для приложения непрерывного мониторинга и обработки заказов.

**Контекст:**
Приложение должно работать как автономный сервис 24/7, мониторя заказы на фриланс-биржах (Upwork, Fiverr, Freelancer.com, PeoplePerHour, Kwork, Weblancer) и обрабатывая их без ручного вмешательства. Приложение ищет заказы, связанные с Viber-ботами и веб-кодированием, поэтому аутентификация должна быть полностью автоматизирована и не требовать ввода кодов подтверждения или двухфакторной аутентификации в ручном режиме.

**План должен включать:**

1. **Анализ API каждой платформы**
   - Для каждой биржи: описание доступных эндпоинтов, методы доступа (public/private), ограничения на количество запросов (rate limits), требуемые permissions, документация

2. **Методы получения и хранения токенов доступа**
   - Способы первичной аутентификации (Client ID/Secret, API ключи, OAuth)
   - Механизм безопасного хранения в переменных окружения
   - Ротация и обновление токенов (где требуется)
   - Управление сроком действия токенов

3. **Обработка OAuth 2.0**
   - Для платформ, требующих OAuth (Upwork, Fiverr): детальное описание Authorization Code Flow
   - Автоматическое обновление refresh-токенов
   - Обработка expired tokens без перерывания сервиса
   - Способ первичной генерации токенов (одноразовая процедура настройки или автоматизация)

4. **Альтернативные методы для платформ без полноценного API**
   - Идентификация платформ, где нужен веб-скрейпинг (Kwork, Weblancer, локальные сервисы)
   - Описание подхода к скрейпингу с соблюдением robots.txt и Terms of Service
   - Рекомендации по технологиям (Selenium, Playwright, BeautifulSoup)
   - Эмуляция браузера и управление сессиями

5. **Безопасность учетных данных**
   - Структура .env файла с примерами
   - Избегание хардкода токенов в коде
   - Защита от утечек (gitignore, логирование без чувствительных данных)
   - Шифрование токенов при хранении в БД (если требуется)

6. **Обработка ошибок и восстановление соединения**
   - Стратегия повторных попыток (exponential backoff)
   - Обработка 401 Unauthorized, 429 Too Many Requests, 503 Service Unavailable
   - Детектирование и восстановление после разрыва соединения
   - Мониторинг состояния аутентификации (healthchecks)
   - Логирование всех ошибок аутентификации с временем и деталями

**Дополнительные требования:**
- План должен учитывать асинхронную природу 24/7 приложения (использование asyncio, connection pooling)
- Описать, как приложение будет работать при перезагрузке сервера или восстановлении после сбоя
- Включить рекомендации по мониторингу и alerting для критических ошибок аутентификации

Структурируй ответ по платформам и по типам задач. Для каждой платформы дай конкретные инструкции; для каждого типа задачи (OAuth, скрейпинг, управление токенами) дай обобщенный подход с примерами кода на Python.
