Ты — опытный архитектор систем с глубокой специализацией в построении отказоустойчивых 24/7 сервисов. На основе плана аутентификации и интеграции API, который уже был разработан, твоя задача — создать комплексную схему обработки ошибок и восстановления соединения для приложения непрерывного мониторинга фриланс-бирж.

**Контекст задачи:**
Приложение работает как автономный сервис 24/7, мониторя заказы на фриланс-биржах и обрабатывая их без ручного вмешательства. Приложение должно найти и обработать заказы, связанные с Viber-ботами и веб-кодированием. Критическое требование: сервис не должен прерывать свою работу при сбоях одной или нескольких бирж — он должен продолжать мониторинг остальных платформ и восстанавливать соединения автоматически.

**План должен включать:**

1. **Классификация типов ошибок аутентификации**
   - Expired tokens (истекшие токены доступа)
   - Invalid credentials (неверные учетные данные, утеря доступа)
   - Rate limits (превышение лимитов запросов)
   - Network failures (сетевые сбои, потеря соединения)
   - Service unavailability (платформа временно недоступна)
   - Authentication mismatch (несовпадение permissions/scope)
   - Для каждого типа ошибки: код ошибки (HTTP status), как она проявляется, чем отличается от других типов

2. **Стратегии автоматического восстановления для каждого типа ошибки**
   - Exponential backoff с джиттером (исходный интервал, максимальный интервал, коэффициент увеличения)
   - Retry logic: сколько попыток до отказа, когда начинать retry (сразу или после задержки), как различать retriable и non-retriable ошибки
   - Для каждого типа ошибки из пункта 1: конкретная стратегия восстановления (паттерн решения)
   - Примеры кода на Python с использованием asyncio

3. **Механизм обновления refresh-токенов без прерывания работы**
   - Когда инициировать обновление (проактивно за N минут до истечения или реактивно при ошибке 401)
   - Как обновлять токен, пока текущие запросы еще обрабатываются (thread-safe или async-safe операция)
   - Механизм блокировки конкурирующих обновлений (чтобы не несколько потоков не обновляли одновременно)
   - Откат и fallback, если обновление refresh-токена тоже закончилось ошибкой
   - Хранение нового токена (если нужно обновлять в БД, в памяти, в .env)

4. **Логирование и мониторинг всех сбоев аутентификации с алертами**
   - Структура логов: timestamp, платформа, тип ошибки, деталь (error message), статус попытки восстановления, успех/неудача
   - Что логировать: само событие ошибки, каждую попытку recovery, финальный результат
   - Что НЕ логировать: чувствительные данные (токены, пароли, полные credentials)
   - Метрики для мониторинга: количество ошибок по типам, время между ошибкой и восстановлением, частота каждого типа ошибки
   - Условия для алертов: N ошибок за M минут, платформа недоступна более X минут, тип ошибки, который указывает на потерю доступа (не recoverable)
   - Каналы для алертов и куда их отправлять (логи, email, webhook, интеграция с мониторингом)

5. **Fallback механизмы для платформ без полноценного API**
   - Для платформ, которые используют веб-скрейпинг (Kwork, Weblancer и др.): как реагировать на сбои скрейпера (изменение структуры сайта, блокировка IP, Cloudflare)
   - Стратегия переключения между альтернативными методами: если основной скрейпер не работает, использовать резервный метод (другая библиотека, другой user-agent, прокси)
   - Кэширование и использование последних известных данных, если новые данные недоступны (как долго хранить, как отмечать "stale data" в приложении)
   - Graceful degradation: если скрейпер платформы сломался, продолжать работать с остальными биржами, но отмечать статус Kwork как "в режиме восстановления"

6. **Процедура graceful degradation, когда одна или несколько бирж становятся недоступны**
   - Как приложение должно себя вести: все ли функции доступны с частичным набором платформ, нужно ли снижать функциональность
   - Уведомление пользователя / администратора о том, что биржа недоступна и как долго это может длиться
   - Приоритизация бирж: если ресурсы ограничены, какие платформы мониторить в первую очередь
   - Как восстанавливать работу: когда платформа снова доступна, как вернуться к нормальному режиму
   - Отслеживание истории недоступности: когда упала, когда восстановилась, как часто это происходит

7. **Тестирование resilience с симуляцией различных сценариев отказа**
   - Сценарии: expired token, invalid token, rate limit 429, сетевой timeout, 503 Service Unavailable, разрыв соединения посередине ответа, медленный ответ, пустой ответ, некорректный JSON
   - Для каждого сценария: как его симулировать (mock, patch, тестовый сервер), что должно произойти (система должна восстановиться), как проверить (assertions, логи, состояние системы)
   - Нагрузочное тестирование: как система ведет себя при одновременных ошибках на нескольких платформах
   - Chaos engineering: случайное введение сбоев в production-like среде и проверка, что система остается стабильной

**Структура ответа:**
- Для каждого типа ошибки (пункты 1-2): таблица с типом, кодом, стратегией восстановления, примерами
- Для механизмов (пункты 3-6): архитектурная схема (диаграмма или описание логики потока), код на Python
- Для тестирования (пункт 7): план тестирования с примерами тестов

**Дополнительные требования:**
- Учесть асинхронную природу приложения (asyncio, connection pooling, concurrent requests)
- Описать, как приложение восстанавливается после перезагрузки сервера (состояние токенов, логов, метрик)
- Конкретные рекомендации по инструментам мониторинга (Prometheus, ELK, DataDog или аналоги)
- Код должен быть production-ready с учетом edge cases и потенциальных race conditions
