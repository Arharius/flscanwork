Ты — опытный архитектор высоконагруженных систем, специализирующийся на построении масштабируемых 24/7 приложений с комплексной работой внешних API и оптимизацией производительности.

Твоя задача — разработать полную архитектуру кэширования и синхронизации данных для 24/7 приложения мониторинга фриланс-бирж, которое интегрируется с Upwork, Fiverr, Freelancer.com, PeoplePerHour, Kwork и Weblancer. Система должна обрабатывать заказы по Viber-ботам и веб-кодированию, минимизируя нагрузку на API при сохранении актуальности данных в реальном времени.

**Контекст:**
Приложение работает непрерывно 24/7, мониторя несколько фриланс-бирж одновременно. Из-за разных rate limits, задержек синхронизации и потенциального дублирования одинаковых заказов на разных платформах нужна умная система кэширования, которая снизит количество API запросов и предотвратит перегрузку. Приложение должно самостоятельно восстанавливаться после сбоев, синхронизировать состояние с источниками и сохранять полную историю для аналитики.

**План должен включать:**

1. **Архитектура многоуровневого кэша**
   - In-memory кэш (например, Python dict с TTL или functools.lru_cache): для высокочастотных операций, локальные данные о текущих заказах
   - Redis: для распределённого кэша между несколькими инстансами приложения, с поддержкой expiration и pub/sub для синхронизации
   - Database (SQL): для персистентного хранения профилей фрилансеров, истории заказов, метаданных о кэшировании
   - Описать иерархию обращений: сначала in-memory → Redis → DB → API
   - Структура ключей кэша для разных типов данных (orders, freelancer_profiles, platform_metadata)

2. **Стратегии инвалидации кэша по типам данных**
   - Заказы: TTL 5-15 минут (актуальность критична, но rate limits ограничивают частоту)
   - Профили фрилансеров: TTL 1-2 часа (редко меняются, но нужны для фильтрации)
   - История и архив: TTL не требуется (персистентные данные)
   - Явная инвалидация при получении webhook-события от биржи
   - Частичная инвалидация: обновление только измененных полей вместо полной перезагрузки
   - Проактивная инвалидация перед истечением TTL (предзагрузка свежих данных)

3. **Дедупликация заказов**
   - Алгоритм идентификации дубликатов: сравнение описания, контактных данных, параметров проекта с использованием нечеткого поиска (fuzzy matching, hash-based comparison)
   - Хранение "canonical order ID" для связи дубликатов между платформами
   - Метаданные о дубликатах: на каких платформах встречается, когда был первый и последний раз встречен
   - Стратегия для пользователя: показывать объединенный заказ или отдельно по платформам

4. **Механизм синхронизации состояния**
   - **Webhooks**: для платформ, поддерживающих их (Upwork, Fiverr) — приём уведомлений о новых заказах, изменениях статуса, истечении deadline
   - **Polling**: для платформ без webhooks или как fallback — проверка каждые N минут с интеллектуальной частотой (реже для стабильных данных, чаще для активных проектов)
   - **Двусторонняя синхронизация**: если пользователь действует через приложение (отправляет заявку, меняет статус), обновить в локальном кэше И отправить на биржу
   - Использование версионирования данных (версия заказа, timestamp последнего обновления) для определения актуальности

5. **Оптимизация частоты API запросов**
   - Батчинг: группировка нескольких запросов в один (если API поддерживает)
   - Умный polling: анализ паттернов активности — реже проверять в ночное время, чаще днём
   - Кэширование результатов фильтрации (категория, язык, бюджет) вместо перезапроса каждый раз
   - Отслеживание лимитов: счетчик оставшихся запросов в час/день, планирование запросов в соответствии с лимитом
   - Дифференцированные стратегии: для критичных данных (активные заказы) — чаще, для архива — редко

6. **Обработка конфликтов синхронизации**
   - Сценарий: пользователь оновил заказ локально, одновременно он изменился на бирже
   - Стратегии разрешения: last-write-wins, merge по полям, версионирование с уведомлением пользователя
   - Логирование конфликтов для анализа
   - Механизм отката (rollback) при критических конфликтах
   - User-friendly уведомления: "Заказ был обновлен на платформе, ваши изменения не применены"

7. **Хранение истории и аналитика**
   - Полная история заказов: когда появился, как менялся статус, когда был выполнен/отклонен/истек
   - Паттерны заказов: анализ популярных тегов, бюджетов, времени появления для Viber-ботов и веб-кодирования
   - Прогнозирование: ML модель для предсказания "похожих" будущих заказов
   - Хранение отклоненных/пропущенных заказов для анализа точности фильтрации

**Технические требования:**
- Описать архитектуру с диаграммами потоков данных (in-memory → Redis → DB → биржи)
- Привести примеры кода на Python: реализация многоуровневого кэша, функции инвалидации, обработки webhook
- Указать используемые библиотеки (redis-py, asyncio, sqlalchemy, aiohttp)
- Рекомендации по конфигурации Redis (persistence, eviction policy)

- Мониторинг: метрики кэша (hit rate, miss rate), задержки синхронизации, размер памяти

Эта архитектура опирается на план аутентификации и интеграции API. Предполагается, что аутентификация уже реализована и обеспечивает надежный доступ к эндпоинтам фриланс-бирж. Система кэширования должна обеспечить эффективное использование этого доступа без перегрузки rate limits.
